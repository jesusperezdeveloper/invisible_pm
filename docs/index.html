Las 6 herramientas, por dentro

Cómo funciona la invocación: Claude solo ve el nombre, la descripción y el esquema de parámetros de cada herramienta. Con eso decide cuál llamar y con qué argumentos; Claude Code ejecuta la llamada (con tu permiso), y el resultado vuelve a su contexto como texto. Por eso las descripciones están escritas en español claro: la descripción ES la interfaz.

listar_reuniones(limite=5)Plaud CLI · Fireflies GraphQL
Por qué existe en el MCP

Es el punto de entrada: sin ella Claude no sabe qué reuniones hay ni sus ids. Todo lo demás (leer, publicar, agendar) parte de aquí. Es también la herramienta de prueba perfecta tras el registro: barata, de solo lectura, imposible de romper nada.

Servicio que usa

Con FUENTE_REUNIONES=plaud: subprocess a plaud files (la CLI gestiona el token OAuth). Con fireflies: query GraphQL transcripts(limit:) con Bearer FIREFLIES_API_KEY.

Parámetros y validación

limite: entero 1–25, por defecto 5. Lo valida pydantic ANTES de tocar la API: limite=99 se rechaza en la puerta. Una herramienta valida; un script confía.

Qué devuelve

Una línea por reunión: • [id] título — fecha (duración). El id es la moneda de cambio: Claude lo copia tal cual para leer_reunion.

Cómo la invoca Claude

"¿Qué reuniones tengo de esta semana?"

listar_reuniones({ "limite": 10 })
leer_reunion(id?)Plaud CLI · Fireflies GraphQL
Por qué existe en el MCP

Es la materia prima del acta: resumen automático + transcripción con quién dijo qué. El id es opcional a propósito: "la última reunión" es el caso del prompt mágico, y así el prompt no necesita conocer ids. Diseñar la firma de la herramienta = diseñar la frase que la dispara.

Servicio que usa

Plaud: plaud file + summary + transcript (3 llamadas CLI). Fireflies: query transcript(id:) — ojo: el argumento es id, no transcriptId (bug real cazado en el ensayo).

Parámetros y límites

id opcional (de listar_reuniones; vacío = la más reciente). La transcripción se trunca a 20.000 caracteres: tu reunión real de 3h13m habría metido 170 KB en el contexto de Claude.

Qué devuelve

Bloques etiquetados: REUNIÓN / FECHA · DURACIÓN / RESUMEN AUTOMÁTICO / TRANSCRIPCIÓN. Etiquetas estables = Claude extrae asistentes, decisiones y tareas sin ambigüedad.

Cómo la invoca Claude

"Lee la última reunión…" (paso 0 del prompt mágico)

leer_reunion({})            // sin id → la última
leer_reunion({ "id": "3ec43a28…" })   // una concreta
guiar_propuesta(id?, tipo="servicios")Plaud · Fireflies + plantilla propia
Por qué existe en el MCP

La tool no redacta: cura. El MCP no tiene LLM dentro — el que razona es Claude. Esta herramienta convierte la reunión en el expediente que un consultor senior le daría a un junior: participantes detectados, contenido completo, plantilla de propuesta por secciones y reglas de qué se puede afirmar y qué no. Claude redacta el borrador encima.

Servicio que usa

La fuente de reuniones activa (Plaud o Fireflies, como leer_reunion) + una plantilla embebida de 8 secciones: contexto, necesidades con cita literal, solución, alcance por fases, cronograma, inversión, supuestos y próximos pasos.

La regla anti-invención

Todo lo que la reunión no diga se marca [TBD: qué falta] — presupuestos, plazos, nombres. Y el borrador termina con las preguntas abiertas que el usuario debe responder para cerrar cada hueco. Propuesta con lagunas visibles > propuesta alucinada.

El bonus: también es un prompt MCP

La misma lógica se expone como prompt del servidor — en Claude Code: /mcp__invisible-pm__propuesta. Frase para la clase: "las tools son las manos del MCP; los prompts, sus recetas". Tercer primitivo MCP en un minuto.

Cómo la invoca Claude

"Prepárame una propuesta a partir de la reunión de ayer con Dental Data"

guiar_propuesta({ "id": "80b4a2e3…", "tipo": "servicios" })
// → expediente + plantilla → Claude redacta el BORRADOR con [TBD]s
//   y cierra con las preguntas para afinar inversión, plazos y decisor
publicar_acta_notion(titulo, contenido_markdown, pagina_padre="Proyectos")Notion REST API
Por qué existe en el MCP

Claude redacta, pero necesita un brazo que archive. La división del trabajo es la lección: el LLM produce el acta en markdown (lo que mejor sabe hacer) y la herramienta la convierte en página de Notion. Ni el LLM toca la API, ni la herramienta redacta.

Servicio que usa

API REST de Notion (Notion-Version 2022-06-28): /v1/search para localizar la página padre por nombre y /v1/pages para crear el acta. Auth: token de integración interna (NOTION_API_KEY) — solo ve páginas compartidas con ella.

El detalle técnico que enseña

Notion no acepta markdown: la herramienta traduce a bloques (encabezados, listas, to-dos, negritas) y los envía en tandas de 100, que es el máximo por petición. Ese "adaptador invisible" es la mitad del valor de un MCP propio.

Qué devuelve / errores típicos

La URL de la página creada (se usa luego en la descripción del evento de Calendar). Error estrella: "No encuentro ninguna página 'Proyectos'" = falta compartir la página con la integración (··· → Conexiones).

Cómo la invoca Claude

Paso 1 del prompt mágico: "Crea el acta en Notion dentro de Proyectos/ReservaFácil…"

publicar_acta_notion({
  "titulo": "Acta — ReservaFácil, 8 jul 2026",
  "contenido_markdown": "## Asistentes\n- Valen\n- Voluntario\n\n## Decisiones\n- MVP sin pagos online…",
  "pagina_padre": "ReservaFácil"
})
actualizar_trello(nuevas_tarjetas, completadas, tablero="ReservaFácil")Trello REST API
Por qué existe en el MCP

Separa decidir de ejecutar: Claude extrae de la conversación qué tareas nacen y cuáles murieron; la herramienta hace el CRUD. Los criterios de aceptación como checklist convierten cada tarjeta en una definición de hecho visible — la diferencia entre "apuntar tareas" y gestionar.

Servicio que usa

API REST de Trello con TRELLO_API_KEY + TRELLO_TOKEN: resuelve el tablero por nombre, crea tarjetas en "Por hacer" (/1/cards), añade el checklist "Criterios de aceptación" (/1/checklists) y mueve a "Hecho" (PUT /1/cards).

Parámetros (estructurados)
  • nuevas_tarjetas: lista de {titulo, responsable, criterios[], vencimiento} — el responsable va al título de la tarjeta
  • completadas: títulos (o parte) → coincidencia parcial, sin exigir el nombre exacto
  • tablero: por defecto "ReservaFácil"
Qué devuelve / errores típicos

Resumen línea a línea (✚ creada / ✔ movida / ⚠ no encontrada) + URL del tablero para la verificación en vivo. Si el tablero no existe, el error lista los tableros disponibles: el error como guía, no como castigo.

Cómo la invoca Claude

Paso 2 del prompt mágico: "…una tarjeta por cada tarea nueva, con los criterios como checklist…"

actualizar_trello({
  "nuevas_tarjetas": [{
    "titulo": "Maqueta de la pantalla de reserva",
    "responsable": "Ana",
    "criterios": ["Se ve el hueco libre de cada profesor", "Máximo 3 toques para reservar"],
    "vencimiento": "2026-07-11"
  }],
  "completadas": ["Pantalla de login"]
})
agendar_reunion(titulo, inicio, duracion_minutos=60, descripcion, invitados)Google Calendar API v3
Por qué existe en el MCP

Cierra el ciclo. La reunión agendada hoy es la que el pipeline procesará el jueves: el PM invisible se realimenta. Y lleva el enlace al acta en la descripción — cada eslabón referencia al anterior.

Servicio que usa

Google Calendar API v3 (events.insert en el calendario primary, zona Europe/Madrid). Auth OAuth de escritorio: autorizar_google.py se ejecuta UNA vez (navegador) y deja token.json; el MCP lo refresca solo, sin volver a preguntar.

Parámetros y validación

inicio en ISO (2026-07-09T12:00) — si Claude manda otra cosa, el error enseña el formato con ejemplo. duracion_minutos 15–480 (default 60). invitados: emails opcionales.

Qué devuelve / errores típicos

El htmlLink del evento — lo que se abre en la verificación en vivo. Sin token.json: el error trae el comando exacto de autorización. La demo degrada con elegancia, no revienta.

Cómo la invoca Claude

Paso 3 del prompt mágico: "Crea en Google Calendar la reunión del jueves a las 12:00 con el enlace al acta…"

  "titulo": "Seguimiento ReservaFácil",

  "duracion_minutos": 60,

})

Prerrequisitos pre-work · alumnos

Lo que cada asistente trae hecho de casa (~25 min). Sin esto solo puede mirar; con esto sale de clase con el sistema funcionando en su máquina.

0/7

0.1 · Herramientas en la máquina

~12 min
Por qué

La clase es 100 % práctica: la terminal es el aula. uv resuelve en un comando el clásico "en mi máquina no funciona el Python".

Si no se hace

El alumno se convierte en espectador. En directo no hay tiempo de depurar instalaciones: se pierde el bloque entero.

0.2 · Cuentas creadas

~13 min
Por qué

El sistema escribe en cuentas reales del alumno: el momento mágico solo emociona si las tarjetas aparecen en SU tablero.

Si no se hace

Los registros con verificación de email en vivo consumen 10-15 min de clase, y sin cuenta de Fireflies el alumno no tiene oídos: su pipeline no arranca.

01

Las llaves — credenciales de las 4 acreditaciones

Cada servicio exige su llave. Sin llave no hay acceso — y eso es una característica de seguridad, no un fastidio. Todas viven en el .env del repo (gitignorado — ver Fase 2).

0/8

1.0 · Plaud — sesión de la CLI (la fuente por defecto)

30 s✓ autenticado 07/07

Los oídos por defecto del MCP son Plaud: decenas de reuniones reales ya procesadas, sin depender de que Fireflies digiera nada a tiempo. La CLI gestiona el token; el MCP la invoca por debajo.

Por qué

Contenido real mata contenido de laboratorio: la demo lee TUS reuniones. Y dos proveedores de oídos son doble red de seguridad.

Si no se hace

El token caducado es el fallo silencioso clásico: en el ensayo apareció caducado. El MCP lo detecta y pide plaud login, pero en directo son 2 min de OAuth delante de todos.

1.1 · Fireflies — API key (fuente alternativa + camino de los alumnos)

2 min✓ hecha en el ensayo
Por qué

Es la llave de tus reuniones: acceso de lectura a transcripciones y resúmenes vía GraphQL. Los alumnos replican este camino (Plaud es hardware).

Si no se hace

El ejercicio del cambio de oídos se queda sin destino y los alumnos sin camino replicable.

1.2 · Trello — key + token

3 min✓ hecha en el ensayo
Por qué

Trello no tiene MCP oficial; nuestro MCP habla con su API REST directamente. La pareja key+token identifica la app y autoriza tu cuenta.

Si no se hace

actualizar_trello falla: el acta se publica pero el tablón se queda desactualizado — exactamente el problema que vendemos resolver.

1.3 · Notion — integración interna

5 min✓ hecha 07/07
Por qué

Las integraciones internas de Notion solo ven las páginas compartidas explícitamente con ellas: mínimo privilegio de serie.

Si no se hace

Sin token: error al publicar. Con token pero sin compartir: "No encuentro ninguna página 'Proyectos'" — el acta no tiene dónde nacer y el paso 1 del prompt mágico muere en directo.

1.4 · Google Calendar — OAuth de escritorio

~15 min⏳ pendiente

El "oficial con fricción": Google no da API keys sueltas para Calendar; exige proyecto de Google Cloud y consentimiento OAuth. Se hace UNA vez y el token se renueva solo.

Por qué

OAuth protege tu agenda: autorización tuya, revocable y con un único permiso (crear eventos). El script separa la autorización (interactiva, una vez) del servidor (headless, siempre).

Si no se hace

agendar_reunion devuelve el error-guía con el comando exacto. La reunión del jueves no se agenda: el cierre del momento mágico queda cojo, y se agenda a mano — delante de todos.

02

El repositorio — nacer bien, desde cero

Toda aplicación nueva empieza igual: un repo vacío, una protección de secretos y un primer commit. Este paso se hace EN la clase, from scratch — el repo del ensayo se borra antes para que el nacimiento sea real.

0/5

2.1 · Crear el repo vacío en GitHub

1 min
gh repo create jesusperezdeveloper/invisible-pm-live --public --clone
cd invisible-pm-live
# (o desde github.com → New repository → git clone)

⚠️ No reutilices el nombre invisible_pm: ese repo es la red de seguridad — guarda la versión probada del MCP (plan B), el tracking del ensayo y esta misma guía (GitHub Pages). Si lo borras para renacerlo en directo, pierdes las tres cosas en el minuto 55. El nacimiento en vivo con otro nombre tiene el mismo efecto didáctico y cero riesgo. Recuerda también copiar el .env del repo del ensayo al nuevo — las llaves no se regeneran, se reutilizan.

Por qué

El proyecto vive versionado desde el minuto 0: cada paso de la clase es un commit que los alumnos pueden reproducir después. El repo ES el material de la clase.

Si no se hace

El código queda en una carpeta suelta: sin historia, sin distribución a alumnos, sin vuelta atrás si algo se rompe en directo.

2.2 · El .gitignore ANTES que ninguna otra cosa

1 min
# .gitignore — la primera línea de defensa
.env
.env.*
token.json
credentials.json
.venv/
__pycache__/
node_modules/
.DS_Store
Por qué

El orden importa: si el .gitignore existe primero, es estructuralmente imposible commitear una llave por despiste. La seguridad por diseño se enseña con el orden de los comandos.

Si no se hace

La API key viaja en el primer push. Hay bots escaneando GitHub en tiempo real: una clave de Fireflies filtrada = cualquiera lee tus reuniones. Toca revocar, regenerar y reescribir historia — en directo, un desastre.

2.3 · El .env con las llaves de la Fase 1

2 min
# .env — las llaves del MCP (NUNCA se commitea)
FIREFLIES_API_KEY=...
TRELLO_API_KEY=...
TRELLO_TOKEN=...
NOTION_API_KEY=ntn_...
# Plaud no necesita clave aquí: usa la sesión de `plaud login`
# Google Calendar tampoco: usa credentials.json + token.json (gitignorados)
Por qué

Un único lugar para todas las llaves, separado del código: el patrón que los alumnos deben llevarse a cualquier proyecto. Verificar con git status convierte la costumbre en reflejo.

Si no se hace

Llaves pegadas en el código o exportadas a mano en cada terminal: el registro del MCP (Fase 4) se vuelve frágil y las claves acaban en el historial de la shell o del repo.

2.4 · Primer commit — el repo respira

1 min
git add .gitignore README.md
git commit -m "invisible_pm: nace el proyecto (gitignore primero)"
git push -u origin main
Por qué

Un primer commit mínimo establece el ritmo: commit pequeño, mensaje que cuenta una historia. Todo lo que Claude Code genere después se versiona paso a paso.

Si no se hace

El primer commit acaba siendo un volcado gigante de 40 ficheros sin narrativa — inservible como material didáctico y peligroso de revisar.

Construir el MCP Bloque 3 · en directo

El giro épico: Claude Code programa su propia acreditación dentro del repo recién nacido. Construir un MCP es describir bien lo que quieres.

0/5

3.1 · El prompt de construcción

1 min + ~10 de generación

Copiado del bloc de notas, jamás escrito en vivo. Mientras Claude trabaja, se narra en lenguaje llano: "está escribiendo la herramienta de listar", "está definiendo qué parámetros acepta".

Crea un servidor MCP en Python con FastMCP en la carpeta invisible-pm-mcp:
el "Project Manager invisible". Herramientas:
1. listar_reuniones(limite) y leer_reunion(id opcional): API GraphQL de
   Fireflies (https://api.fireflies.ai/graphql, Bearer FIREFLIES_API_KEY).
   leer_reunion devuelve resumen, puntos de acción y transcripción con
   el nombre de quien habla; sin id, la última reunión.
2. publicar_acta_notion(titulo, contenido_markdown, pagina_padre): crea la
   página del acta en Notion (API REST, NOTION_API_KEY) y devuelve su URL.
3. actualizar_trello(nuevas_tarjetas, completadas, tablero): tarjetas en
   "Por hacer" con responsable en el título y criterios como checklist;
   mueve a "Hecho" lo cerrado (TRELLO_API_KEY + TRELLO_TOKEN).
4. agendar_reunion(titulo, inicio, duracion_minutos, descripcion): evento
   en Google Calendar (OAuth con token.json).
Usa FastMCP con dependencias gestionadas por uv (pyproject.toml).
Mensajes de error claros en español que digan cómo conseguir cada llave.
Dime al final el comando exacto de claude mcp add para registrarlo.
Por qué

El prompt ES la lección: el público no técnico ve que construir software con IA es especificar con precisión. Cada línea es una decisión de diseño (revisa la sección "Las 5 herramientas, por dentro").

Si no se hace (se improvisa)

Prompts vagos → herramientas vagas. Y teclear 20 líneas en directo son 3 minutos de silencio incómodo y erratas que luego cuestan depuración.

3.2 · Instalar y arrancar limpio

2 min✓ ensayado
Por qué

uv.lock congela versiones: lo probado la víspera es byte a byte lo que corre en clase. Y en un MCP stdio, stdout es el cable: por ahí viaja el JSON-RPC; lo humano va por stderr.

Si no se hace

Dependencias distintas a las ensayadas ("ayer funcionaba"), o un print() en stdout que corrompe el protocolo: el servidor "no conecta" sin error obvio — el fallo #1 de MCPs caseros.

3.3 · Batería de pruebas con llaves REALES

5 min · la víspera✓ hecha 07/07
Por qué

El test "de protocolo" sin llaves da falsa seguridad: valida el envoltorio, no la conversación con la API real.

Si no se hace

Caso real del ensayo: la query usaba transcript(transcriptId:) y pasó todos los tests sin clave. Contra la API real: error. Habría reventado el momento estrella en el minuto 75.

Gotchas cazados en el ensayo (munición para la depuración didáctica)

① Fireflies: el argumento es transcript(id:), no transcriptId — el propio error de la API dice qué espera: 2 min de oro didáctico (a los 5, corta y usa la versión probada). ② Con clave inválida Fireflies responde HTTP 200 + array errors, no un 401. ③ Plaud: los errores de la CLI van marcados con ✗ [CÓDIGO] — no busques "404" en la salida cruda: el serial_number de una reunión puede contenerlo (nos pasó). ④ La validación pydantic rechaza limite=99 antes de tocar la API: enséñalo. ⑤ Notion: el buscador /search tarda en indexar páginas recién creadas — la tool lo salva recorriendo los hijos de las páginas visibles (verificado creando el acta bajo una ReservaFácil de 30 segundos de vida).

05

Montar el escenario de la demo

El proyecto ficticio ReservaFácil (app de reservas para autoescuelas) debe existir en los tres destinos antes de que el prompt escriba en ellos.

0/6

5.1 · El decorado en cada servicio

10 min
Por qué

Un tablón con historia hace la demo creíble: se VE que el sistema entra en un proyecto vivo. Y el acta nueva junto a la antigua muestra continuidad.

Si no se hace

"Mover a Hecho" no tiene nada que mover, el acta cae en una página huérfana, y la audiencia ve un juguete en vez de su martes.

5.2 · El contenido del gancho

10 min · la víspera⏳ pendiente

Con Plaud como fuente por defecto, el gancho del minuto 2 ya tiene contenido garantizado: tus reuniones reales. La pregrabada de ReservaFácil sigue siendo deseable para que la demo hable del proyecto ficticio y no de un cliente real.

Por qué

Fireflies tarda de minutos a más de una hora en procesar. El truco de magia exige una reunión YA digerida; la reunión en vivo del minuto 8 (esa sí, con Fireflies en el Zoom) se procesa durante la clase y se cosecha al final.

Si no se hace

Sin pregrabada de ReservaFácil, el gancho usa una reunión real tuya: funciona, pero expone contenido de clientes en pantalla. Y sin el .txt, si TODO lo que escucha cae, no queda plan B con contenido.

5.3 · El ejercicio del cambio de oídos

3 min · remate del B3✓ verificado

El MCP arranca escuchando por Plaud. En clase se re-registra con -e FUENTE_REUNIONES=fireflies, se reinicia la sesión, y el mismo prompt funciona contra Fireflies.

Por qué

Es la tesis de la clase hecha demo: "he cambiado los oídos sin tocar el archivo, el tablón ni la agenda". Una variable de entorno = un proveedor nuevo. Los alumnos replican el camino Fireflies (Plaud es hardware).

Si no se hace

Se pierde el argumento más contundente a favor del MCP propio: el control sobre las costuras. La clase ve UN pipeline, no una arquitectura.

   como checklist. Si algo se dio por cerrado en la reunión, muévelo a "Hecho".
3. Crea en Google Calendar la reunión acordada para el jueves a las 12:00
   con el enlace al acta en la descripción.
Cuando termines, resume todo lo que has hecho.
Por qué

Leerlo en voz alta lo convierte en guion compartido: la audiencia sabe qué esperar y puede verificar que ocurrió. El E2E de víspera es el único test que ejercita el acta completa markdown→Notion con contenido real.

Si no se hace

Sin E2E previo, la clase ES el primer test de integración del pipeline. Todo está verde por separado; los fallos que quedan viven justo en las costuras.

6.2 · Verificación en vivo + aterrizaje

min 88–108
Por qué

Narrar cada llamada ("acaba de pedir el transcript… ahora escribe en Notion… acreditación por acreditación") es la anatomía del MCP contada en tiempo real. La validación humana no es un descargo: es diseño responsable.

Si no se hace

Sin verificación visible, la demo parece trucada. Sin el matiz del control humano, alguien automatiza actas de clientes el lunes y descubre las alucinaciones el martes.

B

Plan B por capas

Interiorizado, no leído. Regla: 2–3 min de depuración en vivo son oro didáctico; a los 5, se corta y se cambia de capa.
